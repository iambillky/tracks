# DCMS Phased Deployment Plan

## Executive Summary
This document outlines the incremental deployment strategy for the Data Center Management System (DCMS), focusing on delivering immediate value through small, iterative phases.

---

## 🎯 Phase 1: IPAM Foundation (Week 1)
**Goal: Establish network visibility**

### Deliverables
- [ ] Import ARIN allocations from CSV
- [ ] Parse router config for VLAN-to-subnet mappings
- [ ] Create basic database schema for IPRange and VLAN models
- [ ] Display all VLANs with their associated subnets
- [ ] Show total IP count per subnet

### Success Metrics
- Can see all 5 ARIN allocations
- Can see all VLANs with their subnets (including secondaries)
- Can identify which VLANs are for infrastructure vs. colo

### Technical Tasks
```python
# Core models needed:
- IPRange: network, cidr, vlan_id, network_type
- VLAN: vlan_id, name, description, purpose (infra/colo)
- VLANSubnet: vlan_id, subnet, gateway, is_primary
```

## ✅ Phase 4: Validation Layer (Week 4)
**Goal: Prevent configuration errors**

### Deliverables
- [ ] Auto-detect VLAN from IP address
- [ ] Prevent duplicate IP assignments
- [ ] Validate IPs are within defined ranges
- [ ] Enforce Colo* servers use 66.x space
- [ ] Validate all IPs for a server from same subnet
- [ ] Block invalid configurations

### Success Metrics
- Zero invalid IP-to-VLAN mappings
- Cannot assign same IP twice
- Cannot put colo customer on infrastructure IPs
- Form validations prevent all common errors

### Validation Rules
```python
Rules to implement:
1. IP must be within defined ARIN ranges
2. IP determines VLAN (not user choice)
3. Colo* prefix → must use 66.x space
4. All IPs for one server from same subnet
5. No duplicate assignments
```

## 📋 Implementation Notes

### Development Environment
- Start with SQLite for simplicity
- Use Flask development server
- No authentication initially (add later)
- Focus on functionality over UI polish

### Data Migration Strategy
- NO bulk import from legacy system (untrusted data)
- Manual verification and entry
- Build clean dataset from scratch
- Legacy system remains read-only reference

### Critical Success Factors
1. **Each phase must provide standalone value**
2. **No phase should take more than 1 week**
3. **Test with real workflows immediately**
4. **Get team using it by Phase 3**
5. **Document issues for next phase fixes**

### Risk Mitigation
- Keep legacy system running in parallel
- No "big bang" cutover
- Gradual migration as confidence builds
- Rollback plan for each phase

## 📝 Session Handoff Notes

### Key Discoveries from Planning Session
1. **66.x blocks are SWIP'd space for colo only** (protects IP reputation)
2. **VLANs can have multiple subnets** (primary + secondaries due to exhaustion)
3. **Dedicated servers need 5 IPs from same subnet** (not just same VLAN)
4. **Some servers have 100+ IPs** (edge case but must support)
5. **IPMI can be on public or private network** (varies by server)
6. **No VLAN tagging on private network** (flat network)
7. **Colo racks are separate from infrastructure racks** (no mixing)
8. **All colo customers have "Colo" prefix in server ID**

### Technical Decisions Made
- Build breadth-first (all modules minimal, then enhance)
- No data import from legacy system (untrusted)
- Start with IPAM as foundation
- Validate everything (especially IP-to-VLAN)
- Keep it simple (no complex integrations initially)
